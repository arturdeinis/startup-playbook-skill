# Stage 4 — Build a Product Users LOVE

*Sources: L1 (Altman), L7 (Hale), L4 (Cheung), L17 (Rahman).*

**Until you build something users love, nothing else matters.** You can get everything else right and still fail if you miss this. Founders should spend nearly all their time (a) building the product and (b) talking to users; ignore PR, conferences, BD, advisers.

## The core bet: love > like

- Build something a **small number of users LOVE**, not a large number who merely *like* you. It's far easier to expand love than to upgrade lukewarm enthusiasm. If you get this right you can get a lot else wrong; if you miss it, nothing else saves you.
- **Word-of-mouth growth is the signal it's working.** If you're rationalizing a lack of growth ("a big partnership will save us"), that's real trouble.

## Start simple

- **Smallest possible surface area.** Do one thing extremely well (first Google: a textbox + two buttons with the best results; first Facebook; the iPhone). Even if the eventual plan is complex, start with a smaller subset than you think is smallest.
- MVP = emphasis on **VIABLE** (Cheung): the smallest feature set that actually *solves* the problem — not a flat, broken slice.
- Nail **positioning**: a one-liner of functional benefit ("get your place cleaned for $20/hr"), not paragraphs.

## Be fanatical about details

- The word "fanatical" recurs among great founders: quality of small details, the copy explaining the product, and customer support (e.g. PagerDuty on the ticket queue so a night email still gets an hour-response). They feel physical pain when the product sucks and fix it fast.
- **Two qualities** (Hale, Japanese): *atarimae* (taken-for-granted functionality) and *miryokuteki* (enchanting delight). Nail functionality FIRST; only then add wit/delight, or it backfires. Delight can live in copy, docs, and error pages — not just design (MailChimp magazine-style help cut support; Stripe auto-fills your API keys in docs).

## Treat users like a relationship (Hale)

- Growth = the gap between **conversion** and **churn**. Acquiring users = *dating* (first impressions — the origin story they retell; make first email/login/error/first-support-contact memorable). Existing users = *marriage* (long-term).
- Everyone "fights" about the same things (Gottman → support categories): cost/billing, their clients, performance/speed, competition/partnerships. The **Four Horsemen** predict breakup: criticism, contempt, defensiveness, and **stonewalling** (not responding) — stonewalling is a huge early-startup churn cause. **Answer everyone.**
- **Support-Driven Development:** everyone (especially engineers) does customer support. Builders who support their own code give the best support and fix root causes fast (Kayak's red phone on the eng floor; Wufoo's 10 people supported 500k users at 7–12 min response).
- **Direct user exposure makes designs better** (Jared Spool): min. 2 hours every 6 weeks of real-time contact or software decays.
- Close the **knowledge gap** (user knowledge vs. knowledge needed) by *reducing knowledge needed* (docs, tooltips, contextual help), not only by adding features (which widens the gap). One Wufoo docs redesign cut support 30% overnight.
- **Reducing churn 1% == raising conversion 1%**, but it's cheaper/easier and usually neglected. Keep the fire alive (relationships run down): show ongoing value ("Since you've been gone" changelog was Wufoo's most-loved feature); say thank you.
- **Customer intimacy** is one of three paths to dominance (Treacy/Wiersema: best price / best product / best overall solution) — and the only one needing almost no money, just humility and manners.

## The feedback engine (Cheung/Altman)

- **Recruit first users by hand** — don't buy ads yet. Do things that don't scale (see Stage 5).
- Watch them use it. Ask what they'd pay for, whether they'd be bummed if you vanished, what would make them recommend you. Make the loop as tight as possible.
- **Optimize for the next stage only** (10→100, not 10M). Do things manually before automating (learn which signups predict good outcomes). "**Temporary brokenness > permanent paralysis**" — build the generic case, ignore edge cases.
- **Beware the Frankenstein approach** — don't build every requested feature; dig to the underlying problem behind the request. Ship the smallest 1–2 week version of each idea.

## Metrics keep you honest

The company builds whatever the CEO measures. Ignore vanity (total registrations); track **active users, growth, cohort retention, revenue, NPS**. Be brutally honest if they're not going up.

## Just launch

Stealth/perfection assumes imitation is cheaper than innovation — someone will fast-follow anyway, so ship and get the feedback. Marketing/sales is "a tax you pay for not making the product remarkable" (Hale).

## Hardware note (Rahman)

If hardware: it's the full stack (hardware + software + data). Everything is a **system** — design across the whole, not silos. Use phase gates (explore → validate → concept → program → develop → launch → iterate) each with a single decision-maker. Define the **WHYS** (the crisp user problem, such that once solved people can't live without it) + why it matters to *you*. Constraints are good — they force the simplest solution.

## Exit gate → Stage 5

Advance when: a small group of users genuinely loves the product (they'd be bummed if it vanished; they tell friends), you have a tight feedback loop, and you're measuring real engagement/retention — not vanity metrics.
